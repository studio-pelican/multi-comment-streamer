# Implementation Plan: YouTube & TwitCasting Comment Integration Overlay

## 1. 要件理解の要約
- **目的**: YouTube Live と ツイキャスのコメントを1つのWeb画面に統合表示し、OBSのブラウザソースとして利用できるオーバーレイを作成する。
- **対象ユーザー**: 個人配信者、VTuber、YouTubeとツイキャスを同時運用する配信者。
- **MVP機能**: YouTube/ツイキャス両方のコメントを統合取得、プラットフォームごとの判別視覚表現（アイコンや色）、新規コメントのアニメーション表示および自動スクロール。
- **OBS連携**: 1920x1080想定、背景透明、ブラウザソースとしてURLを貼るだけで動作する。
- **非機能要件**: 軽量、低遅延、エラー時の自動再試行、片側停止時ももう一方は継続表示。

## 2. 推奨アーキテクチャ
- **フロントエンド フレームワーク**: Next.js (App Router, TypeScript)
- **スタイリング**: Tailwind CSS (ダークテーマ、カスタムカラー対応)
- **アニメーション**: Framer Motion (新規コメントのポップイン、古いコメントのフェードアウト)
- **データ・状態管理**: React Hooks (`useState`, `useEffect`) による独自キュー管理と `localStorage` による設定保存
- **ホスティング**: GitHub Pages を前提とした Static Export (`next build` + `output: 'export'`)

## 3. 画面構成
1. **初期設定画面 (`/`)**
   - フォーム: YouTube とツイキャスの接続情報、コメント表示件数、各種表示オプション
   - アクション: 「オーバーレイURL生成」「デバッグプレビュー」
2. **オーバーレイ画面 (`/overlay`)**
   - 背景透明（OBS用）
   - コメントの縦長カードリスト
   - 各プラットフォームの色分け・アイコン（例: YouTube=赤系, ツイキャス=青系など）
3. **ステータス表示（オーバーレイ非表示レイヤー等）**
   - APIリクエスト状況や接続エラー状況の簡易確認用インジケーター

## 4. データフロー
1. 設定画面で入力された情報をもとにURLクエリパラメータ（例: `?yt=...&tc=...`）を生成する。
2. OBSブラウザソースでそのURLを開く。
3. マウント完了時、各プラットフォーム専用のコメントポーリング（Hooks）を開始する。
4. 取得した各データは共通の `NormalizedComment` 型へと変換・正規化する。
5. 統合ストア（キュー）へ流し込まれ、到着またはタイムスタンプ順にソートしてリスト表示する。
6. 設定された最大表示件数超過時、または一定時間経過したメッセージはリストから破棄（フェードアウト）する。

## 5. API/取得方式の整理
- **YouTube Live**:
  - 公式の YouTube Data API v3 `liveChatMessages` の使用を基本方針とする。安定・高速だが、API Key が必要である。
- **ツイキャス (TwitCasting)**:
  - 公開API v2の `Get Comments`、あるいは公式のWebSockets接続/Webhookを利用する。もしくは代替手段としてコメント埋め込み等のポーリング。

## User Review Required

> [!IMPORTANT]
> **API取得におけるCORS制約と認証に関する方式確認**
>
> クライアントサイド（GitHub Pagesの完全静的サイト）から直接データ取得する場合、CORS制約の回避や公式APIの認証（API Key等）が必要となります。

- **代替案1 (採用推奨)**:
  ユーザー自身に「YouTube API Key」および「ツイキャスのアクセストークン」を取得させ、初期設定画面に入力してもらう（localStorageに保存）。
  - **メリット**: GitHub Pagesで完全動作。一番安定し規約にも安全。
  - **デメリット**: ユーザー（配信者）の初期設定のハードルがやや高い。

- **代替案2**:
  Next.js の API Routes (サーバー側のプロキシ) を用い、CORSを回避してスクレイピング（例: 配信URLからのDOM解析等）を併用する。
  - **メリット**: ユーザーがAPIキー等を取得・入力する手間が省ける可能性がある。
  - **デメリット**: デプロイ先が GitHub Pages ではなく Vercel または自社サーバー等（Node.js環境）になる。また非公式な取得方法を用いた場合は配信サイト側の仕様変更ですぐ壊れるリスクがある。

**確認事項**:
今回のホスティング要件「GitHub Pages 想定」を優先し、**代替案1（APIキーやトークンをユーザーに入力させる方式）** を基本方針として進めてよろしいでしょうか？あるいは、配信者の使い勝手のためにホスティング要件を変更し、中継サーバー（代替案2など）の導入を検討しますか？
